Chapter 10

Some examples are in the console and code comments.

  • There are three types of errors:
    • System Error: Problem with the system or external devices.
    • Programmer error: Program contains incorrect syntax or logic.
    • User error: User entered data incorrectly
  • Programmers are responsible for minimizing and fixing programmer errors.
  • Programmers should be aware of potential system and user errors to prevent them.
  • Exceptions are errors that produce a return value that can be used to deal with the error. (ex: ReferenceError)
  • Exceptions produce a stack trace, or the sequence of functions that lead to the error.
  • Warnings occur when there is an error that doesn't stop the program from running. (ex: assigning to undeclared variable)
  • Javascript is fairly forgiving when it comes to errors, and will sometimes fail silently.
  • Because Javascript fails silently, programmers should make code that failes loudly.
  • Strict mode allows JavaScript to produce more exceptions and warnings, and prohibits use of deprecated features.
    • Strict mode is used by adding the following as the first line of the js file: 'use strict';
  • Sloppy mode is a term for when strict mode is not used.
  • You can apply strict mode to a function by adding 'use strict'; to the start of the function.
  • Modules do not need 'use strict' because they are in strict mode by default.
  • Linting Tools (JS Lint, JS Hint, ES Lint) highlight sloppy programming practices and syntax errors. They can also enforce a programming style guide.
  • Feature Detection is using an if statement to check whether a feature of a browser exists before trying to call a method.
  • Debugging is the process of finding out where bugs occur in the code.
  • Putting alert() in the code is a basic form of debugging because alerts stop the program from running.
  • Code can be debugged in the console, using console.log(), console.trace(), and other methods.
  • You can also use the console debugging tool to create breakpoints for pausing the code and playing through it.
  • The 'debugger' keyword to create a breakpoint in the code to pause it.
    • The 'debugger' keyword should always be removed before the code is shipped.
  • Error objects can be created by the host environement or in the code using a constructor function.
    • Other Error objects are: RangeError, ReferenceError, SyntaxError, TypeError, URIError, InternalError
    • Safe error object properties across browsers are: name, message, stack.
  • You can throw your own exceptions using the 'throw' statement. It is best to throw an error object.
  • The 'throw' statement can be included in an if statement.
  • When an exception occurs, the program terminates with an error message. This is ideal in development, but not in production.
  • Errors can be hidden from users but still identified.
  • If you think a code will throw an exception, you can wrap it in a try block: try {} catch(error) {} finally {}
    • Resulting errors will be thrown into a catch block.
    • If no error, the program will run.
    • finally blocks are optional, but will always execute after the try or catch block, regardless of an exception.
  • Testing the code is important, you can write tests to ensure that the code works properly.
  • Test-driven development (TDD) means writing the tests first, then writing the code to pass the tests.
  • There are programs to help with test driven development. Not everyone uses TDD though.
